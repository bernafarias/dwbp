This document reports on evidence and implementations of the Data on the Web Best Practices (DWBP) . In particular, it aims to demonstrate that the Best Practices defined in DWBP are already in use and are also implementable.

This is required.

Introduction

One of the main goals of the Data on the Web Best Practices (DWBP) is to facilitate interaction between publishers and consumers of data on the Web. A set of 35 Data on the Web Best Practices was created to cover different aspects related to data publishing and consumption, such as metadata, data licenses, data provenance, data quality, data versioning, data identifiers, data formats, data vocabularies, data access and APIs, data preservation, feedback, data enrichment, and republication. To show that the DWBP are implementable and broadly adopted and referenced, we adopted the following strategies:

  1. We have documented the existence of multiple evidences to show that a given best practice has been adopted by renewed organisations when publishing or consuming data on the Web. An evidence can be of different types, including: datasets; data portals; documents; references; guidelines; good practices.
  2. We have shown that each one of the proposed DWBP is implementable. Therefore, we have documented the existence of at least two implementations demonstrating the feasibility of the DWBP.

Evidence or implementation should cover one or more of the DWBP, regardless the strategy. During the implementation phase we looked for existing evidences as well as new implementations. This information was gathered as follows:

  • DWBP evidence form
  • DWBP template form

This report summarizes both surveys results and implementation details.

Methodology

We considered the following categories to collect evidence for the DWBP:

Datasets, Data Portals and Vocabularies:

This category concerns datasets, data portals and vocabularies available on the Web. TThis type of evidence shows that the DWBP were already considered by organisations responsible for publishing data on the Web. A dataset may be available on a data catalogue implemented using technologies such as CKAN and Socrata, among others. In the same way, a data portal can also be implemented based on one of these technologies. In some cases, proprietary technologies are also used to make the data available on the Web. We also considered vocabularies as an evidence for the DWBP. It is important to note that publishing a vocabulary can be considered as specific case of publishing data on the Web.

Documents and References:

Besides the Datasets, Data Portals and Vocabularies we also considered other types of evidence that could show the relevance of a specific Best Practice.
This category includes Web sites, Web pages, blogs, published papers, APIs documentation and wikis, for example. This type of evidence shows the relevance of adopting a given DWBP and it also works as recommendations.

Guidelines:

We also considered existing Guidelines and forma policies as a type of DWBP evidence. In general, these guidelines were proposed by governments to help data publishers to make data available on the Web. Each guideline discuss and proposes pratices that makes an implicit (or explicit) reference to a DWBP.

Considering the categories above we used the following methodology to collect evidences for the DWBP:

  1. A standard email was sent to several organisations around the world in order to collect evidence.
  2. A review of the evidence was made in order to check which DWBP should have more evidence.
  3. The comments were reviewed among the evidence gathered.
  4. The Implementation Report was elaborated.

Meeting the exit criteria

As noted in the DWBP charter, to move on to Proposed Recommendation, evidence will be adduced in order to demonstrate that each of the best practices has been recommended or adopted in at least two environments, such as data portals and formal policies. Evidence of implementation was gathered from existing datasets, which have already implemented the proposed best practices, as well as from national or sector-specific guidelines that reference the DWBP.

General analysis

One of our main concerns when we started to collect evidence for each one of the DWBP was to have implementations from well-known organisations as well as high profile datasets and data portals worldwide, like DBpedia, Data.gov.uk, Data.gov and World Bank. Analysing the tables presented in the previous section, we can say that we accomplished this goal. The DWBP evidence were collected from well-known organisations and projects including the ones mentioned before as well as BBC, Twitter, Europeana, Pacific Northwest National Laboratory and OpenStreetMaps. Considering the geographical coverage, we collected implementations from several countries, including Brazil, France, Ireland, New Zeland, Spain, UK, USA and Italy. It is also important to notice that evidence in the form of guidelines concerns several governmental organisations from Europe. Other important characteristic from the DWBP implementations is their broad domain coverage, e.g. they refer to different domains, like Government, Environment and Healthcare.

As we can observe in the graphic below, there is a broadly adoption of DWBP related to Metadata (BP1 and BP2), Data Licenses (BP4), Data Identification (BP9 and BP10), Data Formats (BP12 and BP14), Vocabularies (BP15 and BP16), Data Access (BP23, BP24, BP25 and BP26) and Feedback (BP29). On the other hand, for others DWBP, such as Preserve identifiers (BP27), Assess dataset coverage (BP28), Provide real-time access (BP20) and Provide an explanation for data that is not available (BP22), was more difficult to get evidence, specially on the form of datasets and data portals. This can be justified by comments received during the evidence gathering process and also available in the DWBP evidence form. Bill Roberts from the Scottish Government, for example, made the following comment about one of the Data Preservation best practices: "Too difficult to test in a meaningful way. In this system, no datasets have yet been taken offline, so the archiving process has not been developed." In the same way, he made a comment about the Best Practice Provide real-time access: "The system does not currently hold dataset collected in 'real time'. Generally the data is statistical in nature and goes through a slower collection and processing cycle."

DWBP and Data Catalogues

In this section we present some more evidence that shows the adoption of the DWBP. Rather than specific datasets or data portals, we use the following data catalogue solutions as evidence: CKAN, SOCRATA, DKAN, JUNAR, ArcGIS Open Data and OPENDATASOFT. For each one of the DWBP, we show the list of data catalogue solutions that implement it.

BP Data Catalogues Total

As we may notice, there is no evidence for some of the DWBP. This happens because these Best Practices do not concern the solution used for making the data available on the Web, e.g. the data catalogue solution, as explained below.

  • BP10, BP16, BP22, BP28: these BP apply to the data itself rather than the data catalogue solution used to publish the data.
  • BP33, B34, BP35: these BP apply to situations of data republication, e.g, it depends from the consumer rather than the data catalogue solution used to publish the data.
  • BP31: this BP concerns processes that can be used to enhance, refine or otherwise improve raw or previously processed data, which are not part of the basic data catalogue functions.

Concerning BP27....

Some Best Practices related to metadata are partially implemented by the data catalogue solutions. It is important to observe that almost all data catalogue solutions are compatible with DCAT, which means that metadata covered by DCAT are available both in human-readable and machine-readable formats. In general, it means that just a human-readable or a machine-readable version of the metadata is available, as detailed in the following.

  • BP3 is partially implemented by CKAN and JUNAR because they do not offer an explicit way to present human-readable structural metadata.
  • BP4 and BP5 are partially implemented by ARCGIS OPEN DATA because it does not offer a way to represent machine-readable license metadata and machine-readable provenance metadata.
  • BP8 is partially implemented by SOCRATA because it does not offer a way to represent machine-readable version history metadata.

As a general analysis with regards to the Data on the Web Challenges, we can say that Metadata, Data Licenses and Data Formats challenges are a main concern of the data catalogue solutions. The Data Access challenge has also been recognized as an important one except when it concerns real-time data. The use of Data Access APIs is a consensus. The major data catalogue solutions also deal with the Data Identification challenge, however just part of the problem has been solved. The Data Vocabularies challenge has also been considered as an important one since data catalogue solutions reuse existing vocabularies, e.g. DCAT, when publishing metadata about the data catalogues. Other challenges like Data Provenance, Data Versioning and Feedback have been superficially dealt with in the data catalogue solutions. In general, Data Quality, Data Preservation, Data Enrichment and Data Republications are challenges still not explored by the major data catalogue solutions.
